# 当前线段算法与线段 v2.0 差异说明

本文用于核对当前项目实际运行的线段算法，与 `docs/第二讲 笔与线段.pdf` 中“线段 v2.0”口径之间的差异。

结论先行：

- 当前系统默认使用 `seg_algo="chan"`，核心是“基于笔的特征序列线段算法”。
- 当前算法已经覆盖了线段 v2.0 的一部分核心思想：从笔划分线段、用反向笔组成特征序列、在特征序列上找顶底分型、遇到特征序列缺口时等待反向确认。
- 但当前算法并不等价于完整线段 v2.0。它更像“特征序列线段算法 + 尾部未确认段收集 + 若干防误判修正”，不是严格的“线段破坏状态机”。
- 主要差异集中在：特征序列第一/第二元素的包含与缺口处理、缺口后的二次确认边界、中间状态表达、线段只能被线段破坏、复杂重叠区间下的回溯修正。

本文只做规则差异审计，不涉及代码修改。

## 1. 当前系统实际启用的线段算法

### 1.1 默认配置

当前配置入口在 `ChanConfig.py`。

```python
self.seg_conf = CSegConfig(
    seg_algo=conf.get("seg_algo", "chan"),
    left_method=conf.get("left_seg_method", "peak"),
)
```

含义：

- 默认线段算法是 `chan`。
- 默认尾部未确认段处理方式是 `peak`。
- 如果外部没有显式传入 `seg_algo`，系统不会使用 `1+1` 或 `break`，而是使用 `chan`。

`KLine/KLine_List.py` 中的选择逻辑为：

```python
if seg_config.seg_algo == "chan":
    return CSegListChan(seg_config, lv)
elif seg_config.seg_algo == "1+1":
    return CSegListDYH(seg_config, lv)
elif seg_config.seg_algo == "break":
    return CSegListDef(seg_config, lv)
```

其中 `1+1` 和 `break` 在代码里已提示为 deprecated / no longer maintained，所以当前可视化页面和服务端默认应按 `CSegListChan` 理解。

### 1.2 当前线段计算链路

当前项目的走势结构链路是：

```text
原始 K 线
  -> 合并 K 线
  -> 顶/底分型
  -> 笔
  -> 线段 seg_list
  -> 线段的线段 segseg_list
  -> 中枢 / 买卖点
```

线段不直接从原始 K 线或分型上划出来，而是从已经生成的笔上继续划分。

`KLine/KLine_List.py` 中同时维护两类线段：

- `seg_list`：以笔 `CBi` 为输入，得到普通线段。
- `segseg_list`：以普通线段 `CSeg[CBi]` 为输入，再计算更高一级的“线段的线段”。

这意味着：

- 当前普通线段算法不是无限递归算法。
- `segseg_list` 是在普通线段结果之上再做一层结构化，不等同于修改普通线段规则本身。

### 1.3 当前 `chan` 线段算法核心

`Seg/SegListChan.py` 是当前默认算法的核心。

整体流程：

1. `do_init()` 清理尾部不确定线段。
2. 从最后一个确定线段后继续扫描笔。
3. 根据当前疑似线段方向，抽取反向笔作为特征序列。
4. 在特征序列中寻找顶/底分型。
5. 如果特征序列分型可以确认，则生成确定线段。
6. 如果只能看到尾部但没有足够确认，则生成不确定线段。
7. 对剩余不能确认的笔，按 `left_seg_method` 收集成尾部不确定线段。

关键代码：

```python
up_eigen = CEigenFX(BI_DIR.UP)      # 上升线段中抽取下降笔
down_eigen = CEigenFX(BI_DIR.DOWN)  # 下降线段中抽取上升笔
```

含义：

- 判断上升线段是否结束时，抽取下降笔形成特征序列，并寻找特征序列顶分型。
- 判断下降线段是否结束时，抽取上升笔形成特征序列，并寻找特征序列底分型。

这与线段 v2.0 的“反向笔构成特征序列”方向是一致的。

## 2. 线段 v2.0 的规则口径

本节按 `docs/第二讲 笔与线段.pdf` 以及当前页面“线段 v2.0”说明中整理的规则口径描述。

### 2.1 线段的基本定义

线段是由已经确认的笔组成的更高一级走势结构。

基本约束：

- 线段至少由三笔组成。
- 笔方向必须交替。
- 上升线段通常从底部开始，到顶部结束。
- 下降线段通常从顶部开始，到底部结束。
- 后一段的起点应接在前一段的终点上，线段之间不能人为断开。

### 2.2 特征序列

线段 v2.0 的核心不是直接看同方向笔，而是看当前线段内部的反向笔。

```text
上升线段：
  抽取下降笔，组成特征序列；
  在特征序列中寻找顶分型；
  用该顶分型判断上升线段是否结束。

下降线段：
  抽取上升笔，组成特征序列；
  在特征序列中寻找底分型；
  用该底分型判断下降线段是否结束。
```

### 2.3 特征序列的包含处理

线段 v2.0 要求先处理特征序列内部的包含关系，再判断特征序列分型。

但它特别强调：特征序列第一元素与第二元素之间是否存在缺口，是决定线段能否直接确认的重要条件。

也就是说：

- 不能只把特征序列元素当成普通 K 线一样机械合并。
- 如果第一元素和第二元素之间是否有缺口这个信息被合并过程“抹掉”，就可能影响线段结束判断。

### 2.4 第一/第二特征元素无缺口

当特征序列第一元素和第二元素之间没有缺口时：

- 如果后续形成有效特征序列分型，可以直接作为线段结束依据。
- 这种情况下的确认相对直接。
- 本质是：原线段已经被一个有效的特征序列分型确认结束。

### 2.5 第一/第二特征元素有缺口

当特征序列第一元素和第二元素之间存在缺口时：

- 当前特征序列分型不能立即单独确认线段结束。
- 需要等待后续走势给出反向确认。
- 反向确认不足时，最多只能认为进入了疑似结束或中间状态。

这一点是线段 v2.0 里非常关键的特殊处理。

### 2.6 线段只能被线段破坏

线段 v2.0 强调：

> 线段只能被线段破坏。

含义：

- 一根强势笔突破了上一线段关键高低点，并不一定能直接确认上一线段结束。
- 单笔突破只能说明出现了疑似破坏。
- 真正确认破坏，需要后续走势构成反向线段。

这条规则会影响实时行情中的尾部判断：很多看起来“已经突破”的走势，在没有形成反向线段前，仍应保留为未确认或中间状态。

### 2.7 中间状态

线段 v2.0 对复杂走势会保留中间状态。

典型情况：

- 一笔很强，突破了前一线段关键位置。
- 但后续笔没有继续形成明确反向线段。
- 后续走势在前一线段或缺口区间附近反复震荡。

这时既不能简单认为原线段已经结束，也不能确认新线段已经成立。

更准确的表达应是：

```text
原线段疑似被破坏；
但破坏尚未被反向线段确认；
当前处于中间状态。
```

### 2.8 复杂重叠与回溯

线段 v2.0 中还涉及复杂情况：

- 特征序列缺口后，后续又出现与缺口区间重叠的顶/底分型。
- 多个相邻线段的端点可能需要根据后续确认结果回溯调整。
- 同向顶/底比较时，应保留更极端的端点。
- 没被后一线段破坏的线段，理论上应能破坏更前面的线段；如果违反该原则，说明线段划分可能存在问题。

这些规则本质上要求算法具备更明确的状态追踪和回溯校验能力。

## 3. 当前算法与线段 v2.0 的一致点

### 3.1 输入层级一致

两者都不是直接用原始 K 线划线段，而是基于已经形成的笔。

当前系统：

```text
bi_list -> seg_list
```

线段 v2.0：

```text
已确认笔序列 -> 线段
```

这一点一致。

### 3.2 都要求线段至少三笔

当前 `Seg/Seg.py` 中有长度约束：

```python
if end_bi.idx - start_bi.idx < 2:
    self.is_sure = False
```

确定线段的检查里也要求：

```python
if self.end_bi.idx - self.start_bi.idx < 2:
    raise CChanException(...)
```

这表示确定线段至少需要覆盖三笔。

线段 v2.0 同样要求线段至少三笔。

### 3.3 都用反向笔构造特征序列

当前 `CSegListChan.cal_seg_sure()`：

```python
if bi.is_down() and last_seg_dir != BI_DIR.UP:
    up_eigen.add(bi)
elif bi.is_up() and last_seg_dir != BI_DIR.DOWN:
    down_eigen.add(bi)
```

含义：

- 上升线段使用下降笔作为特征序列。
- 下降线段使用上升笔作为特征序列。

这与线段 v2.0 一致。

### 3.4 都在特征序列上寻找分型

当前 `CEigenFX.treat_third_ele()` 中：

```python
self.ele[1].update_fx(self.ele[0], self.ele[2], ...)
fx = self.ele[1].fx
is_fx = (self.is_up() and fx == FX_TYPE.TOP) or (self.is_down() and fx == FX_TYPE.BOTTOM)
```

含义：

- 上升线段寻找特征序列顶分型。
- 下降线段寻找特征序列底分型。

这与线段 v2.0 一致。

### 3.5 都识别特征序列缺口

当前 `Seg/Eigen.py` 中：

```python
if (self.fx == FX_TYPE.TOP and _pre.high < self.low) or \
   (self.fx == FX_TYPE.BOTTOM and _pre.low > self.high):
    self.gap = True
```

含义：

- 特征序列顶分型中，如果中间元素与前一元素之间向上跳空，则标记缺口。
- 特征序列底分型中，如果中间元素与前一元素之间向下跳空，则标记缺口。

线段 v2.0 也强调第一、第二特征元素之间的缺口会影响确认方式。

### 3.6 当前算法已经有缺口二次确认

当前 `CEigenFX.can_be_end()` 中：

```python
if self.ele[1].gap:
    return self.find_revert_fx(...)
else:
    return True
```

含义：

- 无缺口时，特征序列分型可直接确认。
- 有缺口时，不直接确认，而是继续找反向分型。

这与线段 v2.0 的“缺口不能直接确认，需要后续确认”大方向一致。

## 4. 关键差异点

### 4.1 差异一：当前算法不是完整线段 v2.0 状态机

当前系统的 `chan` 算法目标是找到可以确认的线段端点。

它的输出主要是：

```text
确定线段 is_sure=True
不确定线段 is_sure=False
```

线段 v2.0 需要表达的状态更细：

```text
原线段延续
疑似破坏
特征序列无缺口确认
特征序列有缺口等待确认
反向线段确认破坏
中间状态
尾部未确认
回溯修正
```

当前代码没有显式建模这些状态。

影响：

- 当前结果能告诉你“这段是否确定”。
- 但不能完整解释“为什么处于中间状态”。
- 也不能明确区分“单笔疑似破坏”和“反向线段确认破坏”。

### 4.2 差异二：第一/第二特征元素的包含处理可能与 v2.0 不完全一致

当前 `CEigenFX.treat_second_ele()` 会先尝试把第二个特征序列元素并入第一个：

```python
combine_dir = self.ele[0].try_add(bi, exclude_included=self.exclude_included)
if combine_dir != KLINE_DIR.COMBINE:
    self.ele[1] = CEigen(bi, self.kl_dir)
```

这表示：

- 如果第一、第二特征元素存在包含关系，当前算法可能会合并。
- 合并后，原本第一/第二元素之间的结构关系会被改写。

线段 v2.0 口径中，第一元素与第二元素之间是否有缺口，是决定能否直接确认的重要条件。

因此这里存在潜在差异：

```text
当前算法：
  倾向于先按通用合并逻辑处理特征序列元素。

线段 v2.0：
  更强调第一/第二元素的缺口或无缺口关系本身不能被机械合并掩盖。
```

可能影响：

- 某些复杂包含场景下，当前算法可能把原本应该进入“缺口待确认”的结构处理成普通分型。
- 也可能把原本应直接确认的无缺口结构，因合并后重置而延后确认。

这是当前算法与线段 v2.0 最重要的差异之一。

### 4.3 差异三：当前缺口判断发生在特征序列分型成型之后

当前缺口标记在 `CEigen.update_fx()` 中完成。

也就是说，流程是：

```text
先凑出三个特征序列元素
  -> 判断中间元素是否形成特征序列顶/底分型
  -> 如果形成分型，再判断中间元素与前一元素是否有缺口
```

线段 v2.0 更强调：

```text
第一/第二特征元素之间是否有缺口，本身就是分类前提。
```

差异在于：

- 当前算法把缺口作为分型成型后的一个属性。
- 线段 v2.0 把缺口作为线段确认路径的前置分流条件。

影响：

- 当前算法必须先形成一个特征序列分型，才会进入缺口分支。
- 如果某些结构在 v2.0 中应因第一/第二元素缺口提前进入等待确认状态，当前算法可能要等到第三元素和分型出现后才处理。

### 4.4 差异四：当前二次确认是 `find_revert_fx`，不是完整“反向线段破坏”判定

当前有缺口时，会调用：

```python
find_revert_fx(bi_lst, begin_idx=end_bi_idx+2, ...)
```

它做的是：

- 从缺口分型端点后面开始。
- 每隔两笔抽取反向特征序列。
- 查找反向分型。
- 如果找到可确认的反向分型，则认为缺口分型可以确认线段结束。

线段 v2.0 的表述更严格：

```text
线段只能被线段破坏；
缺口后需要等待反向线段确认。
```

两者方向相近，但并不完全等价。

当前 `find_revert_fx` 更像“在后续走势里寻找反向特征序列分型确认”，它没有把“反向线段已经完整成立并破坏原线段”作为独立状态对象记录下来。

影响：

- 当前算法可以在很多场景中得到近似正确结果。
- 但在复杂震荡、重叠区间、多次疑似破坏场景下，可能无法完整表达 v2.0 的确认过程。

### 4.5 差异五：当前 `actual_break` 是防误判修正，不等于“线段只能被线段破坏”

当前 `CEigenFX.actual_break()` 主要用于避免包含处理后的假突破。

核心逻辑：

```python
if self.is_up() and self.ele[2].low < self.ele[1][-1]._low():
    return True
if self.is_down() and self.ele[2].high > self.ele[1][-1]._high():
    return True
```

如果没有实际突破，会继续看后面一两笔是否给出证据；证据不足时将 `actual_break_flag=False`，使结果变成未确认。

这与 v2.0 的精神接近：

- 不轻易确认线段结束。
- 尾部证据不足时保持不确定。

但它仍然不是完整的“线段破坏”模型。

区别：

```text
当前 actual_break：
  检查第三特征元素或后续笔是否有实际突破证据。

线段 v2.0：
  要求由反向线段完成破坏确认。
```

影响：

- 当前算法防止了一部分假分型。
- 但它没有显式要求“破坏者自身必须已经构成线段”。

### 4.6 差异六：当前尾部未确认段由 `left_seg_method` 收集，不是 v2.0 的中间态规则

当前尾部处理在 `Seg/SegListComm.py`。

默认：

```python
left_seg_method = "peak"
```

当后续笔不足以形成确定线段时，当前系统会尝试把尾部笔收集成不确定线段。

例如：

```python
collect_first_seg()
collect_left_seg_peak_method()
collect_left_as_seg()
```

这些逻辑会按最高点或最低点寻找一个尾部不确定段端点。

线段 v2.0 的中间态不是简单“把剩余笔按峰值收集成一段”，而是要表达：

```text
当前原线段是否仍延续？
是否已经疑似被破坏？
是否已经形成反向线段？
是否只是强笔突破但缺少确认？
```

影响：

- 当前图上会出现虚线段。
- 但这个虚线段的含义只是“当前算法收集出来的未确认尾段”。
- 它不一定等价于 v2.0 的“中间状态”。

### 4.7 差异七：当前第一段方向有启发式处理

当前 `CSegListChan.cal_seg_sure()` 中对第一段方向有特殊处理：

```python
if len(self) == 0:
    # 尝试确定第一段方向，不要以谁先成为分形来决定
```

它的作用是避免第一段方向被最早出现的特征序列分型错误决定。

线段 v2.0 对线段方向的要求更偏规则定义：

- 上升线段从底到顶。
- 下降线段从顶到底。
- 起止点需要满足线段结构本身。

当前启发式是工程上的稳定性处理，不是 v2.0 规则本身。

影响：

- 开头几笔走势不完整时，当前算法可能给出一个工程上合理但非严格 v2.0 推导的第一段。
- 这类差异主要出现在图表起始区间。

### 4.8 差异八：当前线段回溯只处理尾部，不是完整历史重算

当前 `do_init()` 会删除尾部不确定线段。

```python
while len(self) and not self.lst[-1].is_sure:
    self.lst.pop()
```

另外，如果最后一个确定线段的特征序列第三元素包含不确定笔，也会回退：

```python
if not self.lst[-1].eigen_fx.ele[-1].lst[-1].is_sure:
    self.lst.pop()
```

这说明当前算法有尾部修正能力。

但线段 v2.0 的复杂场景中，可能涉及：

- 后续反向线段确认后，回头确认前一段结束。
- 多个线段之间的破坏关系传递。
- 某些历史端点因后续确认结果需要调整。

当前算法主要处理尾部不确定段和最后确定段，不是完整全历史回溯状态机。

影响：

- 实时数据更新时，尾部几段会被修正。
- 更深历史上的复杂破坏传递，当前算法未显式建模。

### 4.9 差异九：当前“同向高低比较”分散在多个辅助逻辑里

线段 v2.0 规则中，同向顶/底比较要保留更极端端点：

- 顶与顶比较，取更高的顶。
- 底与底比较，取更低的底。

当前代码里有类似思想，但分散在不同位置：

- `FindPeakBi()` 选择最高/最低尾部笔。
- `CEigen.GetPeakBiIdx()` 从特征序列元素中找峰值笔。
- `CSeg.check()` 检查上升/下降线段起终点高低关系。

这说明当前算法有高低点选择逻辑，但不是以“v2.0 同向端点竞争规则”的形式集中表达。

影响：

- 大多数简单场景下结果一致。
- 复杂场景下，规则解释不够透明，调试时不容易知道某个端点为何被保留或替换。

### 4.10 差异十：当前可视化解释还没有展示完整计算证据

当前页面已经新增了“段划分”和“线段 v2.0”文字说明。

但当前图表上还没有完整展示：

- 当前线段的三个特征序列元素分别是哪几笔。
- 第一/第二特征元素之间是否有缺口。
- 当前线段是否处于缺口待确认。
- 当前是否由 `find_revert_fx` 确认。
- 当前是否只是 `left_seg_method=peak` 收集出的尾部虚线段。
- 当前线段失败或重置原因。

影响：

- 用户看到图上的线段结果，但很难直接复盘每段的判定链路。
- 对照线段 v2.0 时，容易把“当前虚线段”误认为“v2.0 中间态”，或者把“当前确定段”误认为“已由反向线段破坏确认”。

## 5. 场景级差异核对

### 5.1 场景一：特征序列无缺口，形成标准分型

线段 v2.0：

```text
特征序列第一、第二元素无缺口；
后续形成有效顶/底分型；
可以确认当前线段结束。
```

当前算法：

```text
ele[1].gap=False；
actual_break_flag=True；
can_be_end() 返回 True；
生成确定线段。
```

一致程度：高。

风险点：

- 如果第一/第二元素之间存在包含关系，当前合并逻辑可能改变元素边界，导致确认位置与 v2.0 口径不同。

### 5.2 场景二：特征序列第一/第二元素有缺口

线段 v2.0：

```text
不能直接确认；
需要等待后续反向线段或反向确认。
```

当前算法：

```text
ele[1].gap=True；
can_be_end() 调用 find_revert_fx()；
找到反向分型则确认；
找不到但扫描到尾部则返回 None，生成不确定线段。
```

一致程度：中高。

差异：

- 当前找的是反向特征序列分型，不是显式构造“反向线段已经破坏原线段”的完整对象。
- 缺口判断是在特征序列分型形成后发生，不是第一/第二元素出现时就进入状态分支。

### 5.3 场景三：单根强笔突破前段关键高低点

线段 v2.0：

```text
单笔突破不等于线段破坏；
必须等待反向线段确认；
否则只能是疑似破坏或中间状态。
```

当前算法：

```text
actual_break() 会避免部分假突破；
left_bi_break() 可检测最后确定线段后是否有笔突破；
collect_left_seg() 可能收集尾部不确定线段。
```

一致程度：中。

差异：

- 当前没有一个明确字段表示“单笔疑似破坏但未被反向线段确认”。
- 图上通常只表现为虚线段或不确定段。

### 5.4 场景四：强笔后长期震荡在区间内

线段 v2.0：

```text
保持中间状态；
不能过早确认原线段结束；
也不能过早确认新线段成立。
```

当前算法：

```text
若无法形成可确认特征序列分型，会由尾部收集逻辑生成 is_sure=False 线段。
```

一致程度：中低。

差异：

- 当前“不确定线段”不一定等于 v2.0 的“中间状态”。
- 当前缺少中间态分类，例如：疑似破坏、缺口待确认、反向确认不足、仅尾部峰值收集。

### 5.5 场景五：后续分型与前面缺口区间重合

线段 v2.0：

```text
后续顶/底分型即使与前面缺口区间重合，也可能参与成段；
关键看是否满足线段破坏、特征序列确认和端点高低关系。
```

当前算法：

```text
主要通过特征序列合并、reset、find_revert_fx、collect_left_seg 等逻辑处理。
```

一致程度：不稳定，依赖具体笔序列。

差异：

- 当前没有专门的“缺口区间重合但允许成段”规则。
- 如果重合结构被包含合并或 reset 改写，最终端点可能与 v2.0 预期不同。

### 5.6 场景六：尾部实时行情持续刷新

线段 v2.0：

```text
最后一段经常处于未确认状态；
需要随着后续笔继续确认或否定。
```

当前算法：

```text
do_init() 删除尾部不确定线段；
必要时删除最后一个依赖不确定笔的确定线段；
然后重新计算尾部。
```

一致程度：中高。

差异：

- 当前回退主要集中在尾部。
- 线段 v2.0 中更复杂的多段破坏传递没有独立校验层。

## 6. 对当前代码的影响判断

### 6.1 如果只做文档展示

当前页面中的“线段 v2.0” tab 可以继续作为规则说明存在。

但建议明确标注：

```text
当前系统实际输出仍来自 seg_algo=chan。
线段 v2.0 是规则口径说明，不代表当前每一段都完全按 v2.0 状态机生成。
```

这样可以避免用户误以为图上的线段已经完全实现 v2.0。

### 6.2 如果要让当前算法完全贴近线段 v2.0

不建议直接在 `seg_algo="chan"` 上硬改。

更稳妥的方式是新增：

```python
seg_algo = "chan_v2"
```

原因：

- 当前 `chan` 算法已经被中枢、买卖点、图表、报告等后续模块使用。
- 直接改变 `chan` 行为，会影响历史回测和当前图表结果。
- 新增 `chan_v2` 可以让两个算法并行对比，逐步验证差异。

### 6.3 最小增强方向

如果先不重写算法，只增强解释和调试能力，建议增加以下元数据：

```text
CSeg.debug_reason
CSeg.eigen_source
CSeg.eigen_gap_case
CSeg.need_revert_confirm
CSeg.revert_confirmed
CSeg.middle_state
CSeg.left_collect_reason
```

这些字段不改变算法结果，只把当前判断过程暴露出来。

可视化上可展示：

- 当前线段由哪三组特征序列元素确认。
- 是否存在特征序列缺口。
- 是否走了 `find_revert_fx`。
- 是否因为 `actual_break_flag=False` 变成不确定。
- 是否只是 `collect_left_seg` 收集出来的尾部虚线段。

### 6.4 完整改造方向

如果要完整实现线段 v2.0，建议独立做一个状态机。

状态可以包括：

```text
RUNNING：当前线段延续
FX_NO_GAP_CONFIRMED：无缺口特征分型确认
FX_WITH_GAP_PENDING：有缺口，等待反向确认
SUSPECT_BREAK：单笔或局部结构疑似破坏
REVERSE_SEG_CONFIRMED：反向线段确认破坏
MIDDLE：复杂中间状态
UNSURE_TAIL：尾部未确认
```

并把每个状态的输入、输出、回退条件记录下来。

## 7. 建议补充的测试用例

为了验证当前算法与线段 v2.0 的差异，建议准备以下人工笔序列或固定行情样本。

### 7.1 无缺口特征序列确认

目的：

- 验证当前算法和 v2.0 在普通场景下结果一致。

样本特征：

```text
上升线段内部出现三个下降笔；
第一、第二特征元素无缺口；
第二元素形成顶分型；
第三元素有实际突破；
应确认上升线段结束。
```

期望：

- 当前 `chan` 确认线段。
- v2.0 也确认线段。

### 7.2 有缺口但无反向确认

目的：

- 验证缺口不能直接确认。

样本特征：

```text
特征序列第一、第二元素之间有缺口；
随后没有形成反向分型或反向线段；
走势到尾部仍未确认。
```

期望：

### 7.3 有缺口且后续反向确认

目的：

- 验证当前 `find_revert_fx` 与 v2.0 反向确认是否一致。

```text
特征序列有缺口；
后续出现清晰反向结构；
反向结构足以确认前线段结束。
```

期望：

- 当前算法通过 `find_revert_fx` 确认。
- v2.0 通过反向线段确认。
- 两者端点最好一致；如果不一致，需要记录差异原因。

### 7.4 第一/第二特征元素存在包含

- 验证当前合并逻辑是否改变 v2.0 判断路径。

```text
第一、第二特征元素存在包含；
包含前后对是否有缺口、是否形成分型的判断不同。
```

期望：

- 当前算法可能合并并 reset。
- v2.0 应保留第一/第二元素关系用于判断。
- 这是重点差异用例。

### 7.5 单根强笔突破但没有反向线段

目的：

- 验证“线段只能被线段破坏”。

样本特征：

```text
一根强笔突破前段关键高/低；
但后续没有形成完整反向线段。
```

期望：

- v2.0 不能确认破坏，只能中间状态。
- 当前算法可能显示不确定尾段，但不一定能明确表达“单笔疑似破坏”。

### 7.6 缺口区间重叠后又出现分型

目的：

- 验证“后续分型与前面缺口区间重合仍可成段”的场景。

样本特征：

```text
前面存在特征序列缺口；
后续顶/底分型的价格区间与缺口区间重合；
但结构上满足线段 v2.0 的成段条件。
```

期望：

- v2.0 应允许成段。
- 当前算法结果取决于包含合并、reset 和反向确认路径，需要重点对比。

### 7.7 实时尾部反复刷新

目的：

- 验证当前 `do_init()` 的尾部回退是否足够。

样本特征：

```text
最后一段起初是不确定段；
后续新笔出现后，原不确定段被推翻或确认；
最后一个确定段的第三特征元素曾包含不确定笔。
```

期望：

- 当前算法应删除尾部不确定段并重算。
- v2.0 应能解释状态从待确认到确认或否定的变化。

## 9. 总结

当前 `seg_algo="chan"` 与线段 v2.0 的关系可以概括为：

```text
当前算法已经实现了特征序列线段划分的主干；
并且对特征序列缺口做了二次确认处理；
但它还不是完整线段 v2.0。
```

最需要关注的差异是：

1. 第一/第二特征元素的包含处理可能掩盖 v2.0 所需的缺口关系。
2. 当前缺口确认依赖 `find_revert_fx`，不是显式“反向线段破坏”状态。
3. 当前只有 `is_sure=True/False`，缺少中间态、疑似破坏、待确认等细粒度状态。
4. 当前尾部虚线段来自工程收集逻辑，不一定等价于 v2.0 的中间状态。
5. 当前回溯主要处理尾部，不是完整多段破坏传递模型。

因此，如果目标是“解释当前系统为什么这么画”，应继续完善当前 `chan` 算法的调试信息。

如果目标是“严格按线段 v2.0 划分”，建议新增独立 `chan_v2` 算法，并与当前 `chan` 并行验证，而不是直接覆盖现有实现。
