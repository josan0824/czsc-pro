# 线段划分规则说明

本文说明当前项目里“线段”是如何划分出来的。目标读者不是代码开发者，所以会尽量用图形化、生活化的语言解释；最后再补充少量代码依据，方便追溯。

## 1. 先说结论

当前 `main.py` 使用的线段划分规则是：

> 默认的 `chan` 线段算法，也就是“基于特征序列的规则划分法”。

它不是简单递归算法，也不是目前启用的“1+1 终结”算法或“笔破坏”算法。

项目里确实还会计算“线段的线段”，也就是 `segseg`。但这不是把同一个划段过程无限递归下去，而是先把普通线段算出来，再把这些线段当成新的基础元素，继续算更高一层的结构。它更像“把分钟图上的小波段合成更大波段”，不是“递归规则本身”。

## 2. 什么是线段

在这套框架里，走势先被拆成更小的结构：

```text
原始K线 -> 合并K线 -> 分型 -> 笔 -> 线段 -> 中枢 / 买卖点
```

“线段”不是直接从每根 K 线划出来的，而是从“笔”划出来的。

可以这样理解：

- K线像每天、每分钟的价格记录。
- 笔像价格走出的一小段上升或下降。
- 线段则是由多笔组成的一段更完整的走势。

所以线段的意义是：过滤掉更细小的摆动，把多个笔组织成一个更大的波段。

## 3. 当前项目启用的是哪种算法

项目配置里，线段算法由 `seg_algo` 决定。

当前默认值是：

```python
seg_algo = "chan"
```

含义是：使用缠论原文里“特征序列”的思路来划分线段。

项目还保留了另外两种算法：

```text
1. chan  ：基于特征序列计算线段，当前默认使用
2. 1+1   ：都业华版本 1+1 终结算法，代码里已标注不建议使用
3. break ：按线段破坏定义计算线段，代码里也标注不建议使用
```

所以，如果没有显式改配置，当前运行用的是第一种：`chan`。

## 4. “递归划分”和“规则直接划分”的区别

你提到的两种说法可以这样理解。

### 4.1 递归划分

递归划分的直觉是：

```text
先划出一层线段；
再把这些线段当作新的走势；
继续用类似规则再划一层；
必要时继续往上划。
```

这种方式强调“层层往上归并”。

### 4.2 按规则直接划分

按规则直接划分的直觉是：

```text
看已经形成的笔；
按一套明确条件判断某根笔是不是线段终点；
满足条件就确认一段；
不满足就继续等待。
```

这种方式强调“当前这一层按规则判定”，不是无限向上递归。

### 4.3 当前项目属于哪一种

当前普通线段 `seg_list` 的计算属于：

> 按特征序列规则直接划分。

但项目同时也会计算：

```text
segseg_list = 线段的线段
```

这表示：普通线段算完后，再把普通线段当成“更高一级的笔”去计算更高层级线段。

所以更准确的说法是：

> 普通线段本身用“特征序列规则直接划分”；项目额外支持把线段再向上组织成“线段的线段”，但这不是当前普通线段算法本身叫递归。

## 5. 特征序列法的核心思想

“特征序列”这个词听起来很抽象，可以先记住一句话：

> 判断一段上涨是否结束，不看上涨笔本身，而重点看这段上涨过程中出现的下跌笔；判断一段下跌是否结束，则重点看这段下跌过程中出现的上涨笔。

举例说明。

### 5.1 如果当前怀疑是一段上涨线段

一段上涨线段里，价格总体往上走，但中间会夹杂一些回落。

这些回落的笔，就是判断上涨线段是否结束的关键材料。

框架会把这些“下跌笔”拿出来，像看 K 线一样看它们是否形成顶分型。

简单说：

```text
上涨线段中，拿下跌笔组成特征序列；
如果这些下跌笔形成了有效顶分型；
就说明上涨线段可能在这里结束。
```

### 5.2 如果当前怀疑是一段下跌线段

一段下跌线段里，价格总体往下走，但中间会夹杂一些反弹。

这些反弹的笔，就是判断下跌线段是否结束的关键材料。

框架会把这些“上涨笔”拿出来，像看 K 线一样看它们是否形成底分型。

简单说：

```text
下跌线段中，拿上涨笔组成特征序列；
如果这些上涨笔形成了有效底分型；
就说明下跌线段可能在这里结束。
```

## 6. 为什么要看“反向笔”

这是特征序列法最容易让人困惑的地方。

可以用开车转弯来类比：

```text
车一直往前开，不能只看它还在不在前进；
要判断它是不是要转弯，往往要看它反方向摆动的力度。
```

走势也类似。

上涨过程中，如果回调越来越有结构，并且形成了有效顶分型，说明上涨这段力量可能结束。下跌过程中，如果反弹越来越有结构，并且形成了有效底分型，说明下跌这段力量可能结束。

所以线段结束点不是随便找一个最高点或最低点，而是通过“反向笔的结构”确认。

## 7. 项目具体如何确认一条线段

当前 `chan` 算法大致流程如下。

### 第一步：先有笔

线段必须建立在笔的基础上。笔还没有形成，线段就不能可靠计算。

### 第二步：判断当前可能的线段方向

框架会根据已有笔的走向，判断现在可能是在找：

```text
上涨线段的结束
或
下跌线段的结束
```

### 第三步：收集反向笔作为特征序列

如果找上涨线段的结束，就收集下跌笔。

如果找下跌线段的结束，就收集上涨笔。

### 第四步：对特征序列做包含处理

特征序列里的元素也可能互相包含。

所谓包含，可以理解成：

```text
后一根的高低范围被前一根包住；
或者前一根的高低范围被后一根包住。
```

框架会先把这些包含关系处理掉，避免因为局部重叠导致误判。

### 第五步：寻找特征序列分型

处理完包含后，框架会寻找三段结构形成的分型。

上涨线段看顶分型：

```text
中间元素比两边更高
```

下跌线段看底分型：

```text
中间元素比两边更低
```

这个分型一旦有效，就可能成为线段结束证据。

### 第六步：检查是否需要额外确认

框架区分两种情况。

第一种，没有缺口：

```text
特征序列分型成立后，线段可以确认。
```

第二种，有缺口：

```text
如果特征序列第一元素和第二元素之间存在缺口，
不能马上确认线段结束，
还要等待后面出现反向且合法的特征序列分型。
```

这一步是为了避免过早确认线段。

### 第七步：生成实线段或虚线段

如果证据充分，线段会被标记为确定：

```text
is_sure = True
```

图上通常画成实线。

如果证据不足，但当前走势已经需要临时表达一个可能的段，框架会生成虚段：

```text
is_sure = False
```

图上通常画成虚线。

## 8. 什么是虚段

虚段是“暂时推断出来的线段”。

它不是最终确认结果，而是框架在当前 K 线信息不足时，给出的临时结构。

虚段常出现在两个位置：

```text
1. 开头：历史数据截取太短，前面信息不完整。
2. 结尾：最新走势还没走完，证据还不够。
```

虚段后续可能变化、延长、消失或变成实段。

非技术人员可以这样理解：

> 实段像已经盖章确认的结论；虚段像当前根据已有信息画出的草稿。

## 9. 剩余未确认笔如何处理

线段计算时，最后一段往往还没完全确认。

项目配置里有一个参数：

```python
left_seg_method = "peak"
```

它决定那些不能归入确定线段的剩余笔怎么处理。

当前默认是 `peak`，意思是：

> 在剩余笔里找最有代表性的高点或低点，用它组织一个临时线段。

这也是为什么图上尾部经常会看到虚线段。

## 10. 项目里的三种线段算法详解

项目里一共保留了三种线段算法：

```text
1. chan  ：基于特征序列的算法，当前默认使用。
2. 1+1   ：都业华版本 1+1 终结算法，代码保留，但不建议继续使用。
3. break ：按“线段破坏”定义划分，代码保留，但不建议继续使用。
```

这三种算法解决的是同一个问题：

> 给定一串已经形成的“笔”，如何判断哪几笔应该合成一条线段，哪一笔是线段结束点。

为了便于说明，下面会用这种简化符号：

```text
U = 向上笔
D = 向下笔

U1 = 第1根向上笔
D2 = 第2根向下笔
```

还会用价格高低表示笔的强弱。例如：

```text
U3 的高点高于 U1 的高点
```

表示第三根向上笔向上突破得更强。

### 10.1 `chan`：基于特征序列的划分算法

#### 10.1.1 一句话解释

`chan` 算法不是简单看价格涨跌，也不是只看某根笔有没有创新高或新低。

它的核心是：

> 判断一条上涨线段是否结束，要看这条上涨里的下跌笔是否形成了有效顶分型；判断一条下跌线段是否结束，要看这条下跌里的上涨笔是否形成了有效底分型。

这就是“特征序列”。

#### 10.1.2 为什么叫特征序列

一条上涨线段，表面上当然是向上的笔比较显眼。但真正判断它是否结束，关键看里面的反向力量，也就是下跌笔。

所以：

```text
上涨线段 -> 取里面的下跌笔 -> 组成特征序列
下跌线段 -> 取里面的上涨笔 -> 组成特征序列
```

这些反向笔就像“走势要转弯之前露出的痕迹”。

#### 10.1.3 代码中的处理流程

当前默认算法在 `Seg/SegListChan.py` 中。

主要流程是：

```text
1. 删除尾部还没确认的虚线段。
2. 从最后一个确定线段之后开始重新计算。
3. 如果可能是上涨线段，就收集下跌笔作为特征序列。
4. 如果可能是下跌线段，就收集上涨笔作为特征序列。
5. 对特征序列做包含处理。
6. 判断特征序列是否形成顶/底分型。
7. 检查分型是否真的可以结束当前线段。
8. 能确认则生成实线段，不能确认则保留虚线段。
```

对应代码关系：

```text
SegListChan.cal_seg_sure()
- 持续扫描笔
- 把反向笔加入 CEigenFX

CEigenFX.add()
- 收集特征序列第一、第二、第三元素
- 判断是否形成分型

CEigenFX.can_be_end()
- 判断这个分型是否真的可以作为线段结束证据
```

#### 10.1.4 无缺口场景示例

假设现在有一段上涨走势，笔序列简化如下：

```text
U1, D2, U3, D4, U5, D6, U7
```

其中：

```text
U = 向上笔
D = 向下笔
```

因为我们在判断“上涨线段是否结束”，所以重点看下跌笔：

```text
D2, D4, D6
```

这三个下跌笔组成了上涨线段的特征序列。

如果它们形成类似这样的结构：

```text
D2：回调较浅
D4：回调更强，位置更突出
D6：回调减弱
```

用图形理解就是：

```text
D2      D6
  \    /
   D4
```

中间的 D4 成为特征序列里的“顶分型证据”。如果没有缺口，也满足实际突破条件，那么框架就会认为：

```text
上涨线段可以在 D4 对应的位置结束。
```

这个结束点不是 D4 这根下跌笔本身随便决定的，而是由特征序列的分型结构决定。

#### 10.1.5 有缺口场景示例

还是上涨线段，特征序列是：

```text
D2, D4, D6
```

但这次 D2 和 D4 之间出现了明显断开，也就是代码里说的 `gap`。

可以理解成：

```text
D2 的范围和 D4 的范围中间有空档，没有正常衔接。
```

在这种情况下，框架不会因为一个顶分型出现就立刻确认上涨线段结束。

它会继续等后面有没有“反向且合法”的特征序列分型。

用非技术语言说：

> 如果转折证据中间断了一截，不能太草率下结论，要等后面再给一个确认信号。

所以有缺口时，`chan` 算法更谨慎。

#### 10.1.6 下跌线段示例

下跌线段正好反过来。

假设笔序列是：

```text
D1, U2, D3, U4, D5, U6, D7
```

因为我们在判断“下跌线段是否结束”，所以重点看上涨笔：

```text
U2, U4, U6
```

如果这三个上涨笔形成底分型，说明下跌过程中的反弹结构已经给出了转折证据。

框架可能在 U4 对应的位置确认：

```text
下跌线段结束，后面开始进入新的上涨线段。
```

#### 10.1.7 `chan` 算法的优点

```text
1. 更贴近缠论原文的特征序列思想。
2. 不只看简单突破，而是看反向笔结构。
3. 能区分普通分型和有缺口分型。
4. 对未确认尾部会生成虚段，方便展示当前走势。
```

#### 10.1.8 `chan` 算法的不足

```text
1. 规则最复杂，理解成本最高。
2. 确认有滞后，因为需要等待足够笔形成分型。
3. 尾部虚段可能变化，不能把虚段当最终结果。
```

#### 10.1.9 当前项目状态

`chan` 是当前默认算法，也是推荐算法。

如果没有显式配置：

```python
seg_algo = "chan"
```

项目就会使用这一套。

### 10.2 `1+1`：都业华版本 1+1 终结算法

#### 10.2.1 一句话解释

`1+1` 算法比 `chan` 更像一种简化的终结判断。

它关心的是：

> 某一笔附近是否出现了符合“前后两笔关系”的终结结构，如果出现，就把前一笔附近作为线段结束点。

这里的“1+1”可以理解成：

```text
一个当前笔 + 后面一个确认笔
```

当前笔先表现出可能的终结特征，后面一笔再给出确认。

#### 10.2.2 代码中的两种情况

`1+1` 算法在 `Seg/SegListDYH.py` 中。

代码里有两个核心判断：

```python
situation1(cur_bi, next_bi, pre_bi)
situation2(cur_bi, next_bi, pre_bi)
```

它们都在比较三类笔：

```text
pre_bi  = 前面隔两笔的参考笔
cur_bi  = 当前正在观察的笔
next_bi = 后面隔两笔的确认笔
```

也就是说，它不是只看当前一笔，而是看：

```text
前面的参考位置 + 当前笔的位置 + 后面的确认笔
```

#### 10.2.3 situation1 的直观解释

以“当前笔是下跌笔”为例。

代码条件大致是：

```text
当前下跌笔的低点，没有跌破前面参考笔的低点；
后面的确认笔，整体又比当前下跌笔弱；
于是认为这里出现了一种终结结构。
```

用简化图理解：

```text
前参考笔低点：100
当前下跌笔低点：105
后确认笔高低都弱于当前笔
```

这说明什么？

```text
下跌没有真正打穿前面的关键低点，
后面又继续表现出结构性转弱，
当前区域可能是一个段落切换点。
```

对上涨笔则完全反过来：

```text
当前上涨笔的高点，没有突破前面参考笔的高点；
后面的确认笔整体又比当前上涨笔强；
于是可能确认线段终结。
```

#### 10.2.4 situation2 的直观解释

situation2 比 situation1 更强调“突破后被确认”。

以“当前笔是下跌笔”为例。

代码条件大致是：

```text
当前下跌笔的低点，跌破了前面参考笔的低点；
后面的确认笔继续处在较弱位置；
于是认为线段终结结构成立。
```

用简化图理解：

```text
前参考笔低点：100
当前下跌笔低点：95
后确认笔低点仍低于前参考笔低点
```

这表示：

```text
当前笔已经产生破坏，
后面一笔没有把这个破坏修复掉，
所以线段可以被终结。
```

#### 10.2.5 `1+1` 算法示例

假设有一串笔：

```text
U1, D2, U3, D4, U5, D6, U7
```

现在观察 D4。

算法会拿它和前面的 D2、后面的 D6 一起比较。

如果满足某种终结结构：

```text
D4 与 D2 的低点关系满足条件；
D6 又进一步确认这个结构；
```

那么代码里会执行：

```text
add_new_seg(..., idx-1)
```

也就是在当前观察笔附近确认一条线段。

注意这里不是像 `chan` 那样构造完整特征序列分型，而是用更直接的前后笔条件来判断。

#### 10.2.6 `1+1` 的虚段处理

`1+1` 算法还有一个尾部处理：

```python
try_update_last_seg()
cal_bi_unsure()
```

大意是：

```text
如果最后一段后面又出现更极端的同向笔，
就把最后线段的终点临时更新过去；
如果后面突破了最后确定线段的尾部，
就尝试生成一个未确认虚段。
```

这就是为什么它也会出现虚线段。

#### 10.2.7 `1+1` 的优点

```text
1. 规则比特征序列简单。
2. 判断更直接，看前后几笔关系即可。
3. 对某些教学体系或人工划线习惯更接近。
```

#### 10.2.8 `1+1` 的不足

```text
1. 当前项目已经标注 deprecated，不再维护。
2. 没有 `chan` 的特征序列处理完整。
3. 对复杂走势、包含关系、缺口场景的解释力弱一些。
```

#### 10.2.9 当前项目状态

如果配置：

```python
seg_algo = "1+1"
```

项目会使用 `CSegListDYH`。

但代码会提示：

```text
Please avoid using seg_algo=1+1 as it is deprecated and no longer maintained.
```

所以它是保留实现，不是当前推荐实现。

### 10.3 `break`：按线段破坏定义划分

#### 10.3.1 一句话解释

`break` 算法最直接。

它的核心是：

> 如果某根向上笔比两笔之前的向上笔更高，就认为出现了向上破坏；如果某根向下笔比两笔之前的向下笔更低，就认为出现了向下破坏。连续出现方向切换的破坏，就用前一个破坏点划出线段。

它更像“看突破”。

#### 10.3.2 为什么是和“两笔之前”比较

笔的方向是交替的。

例如：

```text
U1, D2, U3, D4, U5
```

向上笔之间不是相邻的，而是隔着一根下跌笔：

```text
U1 和 U3 比
U3 和 U5 比
```

所以代码里会拿当前笔和 `idx - 2` 的笔比较。

#### 10.3.3 代码里的核心判断

`break` 算法在 `Seg/SegListDef.py` 中。

两个核心函数是：

```python
is_up_seg(bi, pre_bi)
is_down_seg(bi, pre_bi)
```

含义非常直接：

```python
is_up_seg:
当前向上笔的高点 > 前一个同向向上笔的高点

is_down_seg:
当前向下笔的低点 < 前一个同向向下笔的低点
```

翻译成非技术语言就是：

```text
向上创新高，算向上破坏；
向下创新低，算向下破坏。
```

#### 10.3.4 `break` 算法流程

它会按顺序扫描所有笔。

流程可以理解为：

```text
1. 从第三笔开始看，因为要和两笔之前的同向笔比较。
2. 如果当前向上笔创新高，记录一个向上破坏点。
3. 如果当前向下笔创新低，记录一个向下破坏点。
4. 如果新破坏点和上一个破坏点方向相反，并且中间间隔足够，就确认上一处为线段终点。
5. 如果最后只找到一个破坏点，但还没等到反向确认，就生成虚段。
```

#### 10.3.5 上涨破坏示例

假设有笔：

```text
U1, D2, U3, D4, U5
```

高点如下：

```text
U1 高点 = 100
U3 高点 = 105
U5 高点 = 103
```

当看到 U3 时：

```text
U3 高点 105 > U1 高点 100
```

这就是向上破坏。

算法会把 U3 记为一个潜在线段终点候选。

如果后面又出现向下破坏，比如：

```text
D6 低点 < D4 低点
```

那么说明方向切换了，之前的向上破坏点就可以成为上一条线段的结束点。

#### 10.3.6 下跌破坏示例

假设有笔：

```text
D1, U2, D3, U4, D5
```

低点如下：

```text
D1 低点 = 100
D3 低点 = 95
D5 低点 = 97
```

当看到 D3 时：

```text
D3 低点 95 < D1 低点 100
```

这就是向下破坏。

如果后面出现向上破坏，例如：

```text
U6 高点 > U4 高点
```

算法就认为方向切换，前面的向下破坏点可以作为线段终点。

#### 10.3.7 `break` 算法中的 peak_bi

代码里有一个变量：

```python
peak_bi
```

它可以理解为：

> 当前暂时找到的“最有可能成为线段终点的突破笔”。

如果后面同方向继续创新高或创新低，`peak_bi` 会更新到更极端的位置。

例如上涨过程中：

```text
U3 创新高
U5 又创新高
```

那么终点候选会从 U3 更新到 U5。

#### 10.3.8 `break` 的优点

```text
1. 最直观，容易理解。
2. 主要看创新高、创新低。
3. 计算逻辑相对简单。
```

#### 10.3.9 `break` 的不足

```text
1. 对特征序列、包含、缺口等处理不如 chan 细。
2. 在复杂震荡走势里可能过于机械。
3. 当前项目同样标注 deprecated，不建议使用。
```

#### 10.3.10 当前项目状态

如果配置：

```python
seg_algo = "break"
```

项目会使用 `CSegListDef`。

但代码会提示：

```text
Please avoid using seg_algo=break as it is deprecated and no longer maintained.
```

所以它也是保留实现，不是当前推荐实现。

### 10.4 三种算法对比

| 算法 | 当前状态 | 核心依据 | 直观理解 | 复杂度 | 推荐程度 |
| --- | --- | --- | --- | --- | --- |
| `chan` | 默认使用 | 特征序列顶底分型 | 看反向笔是否形成可靠转折结构 | 高 | 推荐 |
| `1+1` | 保留但不维护 | 前后几笔形成终结结构 | 当前笔 + 后续笔确认终结 | 中 | 不推荐 |
| `break` | 保留但不维护 | 同向笔创新高/新低 | 看突破和反向突破 | 低 | 不推荐 |

### 10.5 用同一段走势对比三种算法

假设有一段笔序列：

```text
U1, D2, U3, D4, U5, D6, U7, D8, U9
```

价格表现大致是：

```text
U1 高点 100
U3 高点 110
U5 高点 108
U7 高点 112
U9 高点 106

D2 低点 90
D4 低点 94
D6 低点 92
D8 低点 88
```

不同算法会关注不同事情。

`chan` 会问：

```text
上涨过程中，那些下跌笔 D2、D4、D6、D8 是否组成了有效特征序列分型？
有没有缺口？
有没有实际突破？
这个分型是否足以确认上涨线段结束？
```

`1+1` 会问：

```text
当前观察笔和前面参考笔、后面确认笔之间，是否满足 situation1 或 situation2？
是否出现了 1+1 终结结构？
```

`break` 会问：

```text
U3 是否比 U1 创新高？
U7 是否比 U5 创新高？
D8 是否比 D6 创新低？
如果先有向上破坏，再有向下破坏，就用前一个破坏点划段。
```

所以三者不是谁更“绝对正确”，而是观察走势的角度不同：

```text
chan  更重视结构证据；
1+1   更重视前后笔终结形态；
break 更重视直接突破。
```

当前项目默认选择 `chan`，是因为它更贴近框架原本的缠论特征序列设计。

## 11. 图上绿色线段应该如何理解

图上的绿色粗线就是 `seg_list` 里的普通线段。

它的含义是：

```text
框架按当前配置，把一串笔组织成一个更大走势段。
```

如果是实线，表示线段已确认。

如果是虚线，表示线段只是当前推断，后续可能变化。

## 12. 最终总结图表：三种线段算法对比

| 对比维度 | `chan` 特征序列算法 | `1+1` 终结算法 | `break` 线段破坏算法 |
| --- | --- | --- | --- |
| 当前项目状态 | 默认使用 | 保留实现，但已提示不建议使用 | 保留实现，但已提示不建议使用 |
| 代码实现 | `Seg/SegListChan.py` | `Seg/SegListDYH.py` | `Seg/SegListDef.py` |
| 核心观察对象 | 当前线段里的反向笔组成的特征序列 | 当前笔、前面参考笔、后面确认笔 | 当前笔和两笔之前的同向笔 |
| 最关心的问题 | 反向笔是否形成有效顶/底分型 | 是否出现 1+1 终结结构 | 是否创新高或创新低，之后是否被反向破坏 |
| 上涨线段如何结束 | 看上涨过程中的下跌笔是否形成有效顶分型 | 看相关下跌/上涨笔组合是否满足 `situation1` 或 `situation2` | 先出现向上破坏，再等后续向下破坏确认 |
| 下跌线段如何结束 | 看下跌过程中的上涨笔是否形成有效底分型 | 看相关上涨/下跌笔组合是否满足 `situation1` 或 `situation2` | 先出现向下破坏，再等后续向上破坏确认 |
| 是否处理包含关系 | 是，特征序列元素会处理包含 | 没有 `chan` 那样完整的特征序列包含处理 | 主要看同向高低点突破，处理较直接 |
| 是否处理缺口 | 是，有缺口时需要额外确认 | 不以特征序列缺口为核心 | 不以特征序列缺口为核心 |
| 线段确认速度 | 较慢，需要等结构证据完整 | 中等，依赖前后笔确认 | 较快，突破关系明显时容易确认 |
| 解释难度 | 高，需要理解特征序列 | 中，需要理解 1+1 终结结构 | 低，创新高/新低最直观 |
| 对复杂走势的适应性 | 较强 | 中等 | 较弱，震荡中可能偏机械 |
| 图上虚段来源 | 证据不足但需要表达当前尾部结构 | 最后一段继续延伸或突破后生成临时段 | 最后只有候选破坏点但缺少反向确认 |
| 适合谁 | 想贴近缠论原文特征序列划段的人 | 熟悉都业华 1+1 教学体系的人 | 想用简单突破规则快速划段的人 |
| 项目建议 | 推荐 | 不推荐作为新分析默认算法 | 不推荐作为新分析默认算法 |

更简洁地说：

| 算法 | 一句话记忆 |
| --- | --- |
| `chan` | 看反向笔组成的特征序列，等可靠分型确认线段结束。 |
| `1+1` | 看当前笔前后的组合是否形成终结结构。 |
| `break` | 看同向笔是否创新高/新低，再看是否被反向突破确认。 |

如果是当前项目的默认分析结果，应优先按 `chan` 理解。

## 13. 一句话总结给非技术人员

当前项目不是简单地把价格上涨下跌递归合并成线段。

它的默认做法是：

> 先把走势拆成一笔一笔，再观察每段走势中的反向笔，把这些反向笔当成一组“特征序列”，看它们是否形成可靠的顶底分型；如果证据充分，就确认一条线段；证据不足时，只画临时虚段。

这就是当前项目使用的“基于特征序列的规则划分法”。
